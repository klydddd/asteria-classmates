# Phase 3 Transcript Normalization and Review Design

## Summary

Phase 3 operates on the `metadata.csv` generated by the Phase 2 PLD importer.
It adds deterministic transcript normalization and a resumable terminal review
workflow. Generic transcript manifests, automatic audio playback, and web
review interfaces remain outside this iteration.

The final dataset builder in Phase 4 will consume only rows whose
`quality_status` is `approved`.

## Commands and Data Flow

`bosesph normalize-transcripts DATASET` reads `DATASET/metadata.csv`, validates
the complete file, normalizes every transcript, and atomically replaces the
metadata file. It also writes `DATASET/normalization_report.json` containing
before/after text, applied rules, and unresolved warnings.

`bosesph review DATASET` processes `pending` and `needs_review` rows in metadata
order. It displays the audio path, transcript, language, speaker metadata,
duration, existing notes, and the required review checklist. The reviewer can
approve, mark needs fix, reject, skip, or quit.

Each review decision is saved by atomically rewriting `metadata.csv`.
Approved and rejected rows are skipped on subsequent runs. Needs-fix decisions
use the existing `needs_review` metadata status.

## Normalization Rules

Safe automatic normalization performs these operations:

- Unicode NFC normalization;
- trimming outer whitespace and collapsing internal whitespace;
- removing spaces before `.`, `,`, `?`, and `!`;
- adding one following space where punctuation separates text;
- collapsing repeated terminal punctuation to one mark;
- canonicalizing recognized annotation casing;
- uppercasing the first cased character of each sentence without lowercasing
  any other text; and
- removing control and zero-width characters.

The normalizer never translates, respells Kapampangan, removes repetitions,
alters code-switching, strips meaningful diacritics, or invents annotations.
Malformed brackets, unsupported annotations, or ambiguous symbols are not
silently rewritten.

Structurally invalid metadata and empty transcripts abort normalization before
publication. Safe text changes preserve the current status. Unresolved text
warnings set the row to `needs_review` and append a deduplicated warning to
`reviewer_notes`.

## Review Rules

The reviewer is asked to confirm audio understandability, transcript match,
language correctness, and speaker metadata completeness before choosing one
final disposition.

- `approve` writes `approved`;
- `needs fix` writes `needs_review` and requires a non-sensitive note;
- `reject` writes `rejected` and requires a non-sensitive note;
- `skip` leaves the row unchanged; and
- `quit` exits normally after preserving prior decisions.

Audio is not played by the command. The resolved path is shown for use with an
external player. A missing audio file cannot be approved.

## Failure and Exit Behavior

Normalization returns `0` when no unresolved warning remains, `1` when the run
completed with transcript warnings, and `2` for unreadable or structurally
invalid input. Existing files remain untouched if publication fails.

Review returns `0` on completion or normal quit and `2` for invalid dataset
input. Keyboard interruption preserves decisions already checkpointed.

Normalization is idempotent: rerunning it does not alter normalized text,
duplicate notes, or duplicate report rules.

## Verification

Tests cover each normalization rule, Unicode and annotation handling,
idempotence, warnings, atomic publication, all reviewer actions, required
notes, resume behavior, missing audio, CLI exit codes, and a synthetic
Phase 2-to-Phase 3 workflow.
